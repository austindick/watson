# Watson Library: Book Schema

## Overview

The Watson library uses a 3-level hierarchy:

```
BOOK.md (index)
  └── CHAPTER.md (content or page index)
        └── PAGE.md (granular content, e.g., one component per page)
```

**Rules:**
- Max 3 levels. No deeper nesting.
- BOOK.md is always an index — it never contains body content. Content lives in chapters (or pages within chapters), never in BOOK.md itself.
- Even simple books have at least one chapter. The Librarian always navigates BOOK.md → CHAPTER.md; it never checks whether content lives in the book file.
- CHAPTER.md may contain content directly (two-level book) or a page manifest (three-level book). Design system books typically use all three levels (one page per component).

---

## BOOK.md Schema

### Frontmatter

```yaml
---
type: book                          # Required. Always "book".
title: string                       # Required. Human-readable book name (e.g., "Design System").
last_updated: string                # Required. ISO date string (e.g., "2026-03-29").
source_paths: string[]              # Required. Paths the Librarian scanned to generate this book.
chapters:                           # Required. Ordered chapter manifest.
  - id: string                      # Required per entry. Unique chapter identifier (kebab-case).
    path: string                    # Required per entry. Relative path from BOOK.md to CHAPTER.md.
    summary: string                 # Required per entry. One-line description for routing decisions.
use_when: string                    # Required. Routing guidance — tells agents when to request this book.
source_hash: string                 # Required. sha256 hash of source content for Librarian freshness detection.
---
```

### Markdown Body

The BOOK.md body contains:
1. A title heading (`# {title}`)
2. A purpose paragraph (what this book covers, who generated it)
3. A chapters table (Chapter | Path | Summary)

**Example:**

```markdown
---
type: book
title: Design System
last_updated: "2026-03-29"
source_paths:
  - src/components/design-system
  - src/styles/tokens
chapters:
  - id: global-theme
    path: global-theme/CHAPTER.md
    summary: CSS custom properties for colors, spacing, radii, and typography
  - id: components
    path: components/
    summary: Component library — props, variants, states, and usage
use_when: Agent needs component props, design tokens, or typography presets
source_hash: sha256:e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855
---
```

## CHAPTER.md Schema

### Frontmatter

```yaml
---
type: chapter                       # Required. Always "chapter".
title: string                       # Required. Human-readable chapter name.
last_updated: string                # Required. ISO date string.
pages:                              # Optional. Only present if chapter delegates to pages.
  - id: string                      # Required per entry. Unique page identifier (kebab-case).
    path: string                    # Required per entry. Relative path from chapter directory to PAGE.md.
---
```

### Markdown Body

- **If the chapter has no pages:** The body contains the chapter's actual content (token tables, usage notes, etc.).
- **If the chapter uses pages:** The body contains a page index table (Page | Path | Summary) and no content itself.

**Example (with pages — design system components chapter):**

```markdown
---
type: chapter
title: Components
last_updated: "2026-03-29"
pages:
  - id: button
    path: button/PAGE.md
  - id: badge
    path: badge/PAGE.md
  - id: input
    path: input/PAGE.md
---

# Components

Component library reference. Each component has its own page with props, variants, states, and usage examples.

## Pages

| Component | Path |
|-----------|------|
| Button | button/PAGE.md |
| Badge | badge/PAGE.md |
| Input | input/PAGE.md |
```

**Example (without pages — simple chapter with inline content):**

```markdown
---
type: chapter
title: Global Theme
last_updated: "2026-03-29"
```

## PAGE.md Schema

### Frontmatter

```yaml
---
type: page                          # Required. Always "page".
title: string                       # Required. Human-readable page name (e.g., "Button Component").
last_updated: string                # Required. ISO date string.
---
```

### Markdown Body

The body contains the page's granular content. For design system component pages, use the canonical template below.

### Component PAGE.md Template

The Librarian uses this exact structure for every component page. Sections with no applicable content are omitted (not left empty).

```markdown
---
type: page
title: {ComponentName} Component
last_updated: "{ISO date}"
---

# {ComponentName} Component

{One-sentence description of the component's purpose.}

**Import:** `import { {ComponentName} } from "{import-path}"`

## Props

| Prop | Type | Default | Required | Description |
|------|------|---------|----------|-------------|
| ... | ... | ... | ... | ... |

## Variants

{List or table of variants with visual descriptions and use_when guidance.}

## States

- **{state}** — {description including which tokens are used}

## Slots / Composition

{Document children prop, render props, or named slots. Omit if none.}

## Responsive Behavior

{Document responsive props or breakpoint variants. Omit if none.}

## Gotchas

{Known quirks, TODOs, or non-obvious behaviors. Omit if none.}
```

**Rules:**
- No JSX code examples — agents generate code from props/variants/states, not from copied snippets
- Every variant row includes the design tokens it uses (e.g., `--slate-color-action-surface-default`)
- States include token references, not just descriptions

## Navigation Pattern

Library navigation is split between **subskills** (book selection) and **agents** (content consumption).

### Subskill responsibility (book resolution)

Subskills read LIBRARY.md and BOOK.md manifests to resolve which chapters an agent needs:

```
1. Subskill reads library/LIBRARY.md
2. Subskill selects relevant book(s) using use_when guidance
3. Subskill reads {book}/BOOK.md chapters[] manifest
4. Subskill resolves specific chapter/page paths based on agent's role
5. Subskill passes resolved paths as libraryPaths[] to agent
```

### Agent responsibility (content consumption)

Agents receive pre-resolved paths and read them directly. Agents **never** read LIBRARY.md or BOOK.md — they don't navigate the hierarchy.

```
1. Agent receives libraryPaths[] — an array of resolved chapter/page paths
2. Agent reads each path directly (e.g., global-theme/CHAPTER.md)
3. If a chapter uses pages: agent reads specific PAGE.md files as needed
```

### Why this split

Subskills run in the orchestrator's context window — book selection is lightweight coordination (read a 46-line index, pick paths). Agents get fresh context windows for heavy work — they shouldn't spend tool calls navigating a hierarchy they don't control.

## LIBRARY.md Schema

LIBRARY.md is the mandatory agent entry point. Agents **NEVER** glob through `library/` — they always start from LIBRARY.md and navigate down. The Librarian auto-updates LIBRARY.md after every generate or update run.

### Frontmatter

```yaml
---
last_updated: string    # Required. ISO date string of the most recent Librarian run.
books_count: number     # Required. Total number of books in the library.
---
```

### Per-Book Entry Format

Each book is documented as a level-3 heading section under `## Books`:

```markdown
### {Book Title}
**Path:** library/{book-dir}/BOOK.md
**Book type:** source-derived | foundational
**Last updated:** {ISO date}
**Chapters:** {comma-separated chapter titles, with counts where relevant}
**Use when:** {routing guidance — when should agents request this book?}
**Last run:** {e.g., "4 chapters, 35 pages, 0 skipped"}
```

### Example (Populated LIBRARY.md)

```markdown
# Watson Library

**Last updated:** 2026-03-29
**Books:** 2

## Books

### Design System
**Path:** library/design-system/BOOK.md
**Book type:** source-derived
**Last updated:** 2026-03-29
**Chapters:** Global Theme, Components (35 components), Icons, Patterns
**Use when:** Agent needs component props, design tokens, typography presets, or icon references
**Last run:** 4 chapters, 35 pages, 0 skipped

### Playground Conventions
**Path:** library/playground-conventions/BOOK.md
**Book type:** foundational
**Last updated:** 2026-03-29
**Chapters:** Project Structure, Scaffolding, Components, Design Tokens, Dev Workflow, Multi-Variant, Contributor Registration
**Use when:** Agent needs prototype scaffolding steps, component imports, design tokens, or dev workflow
**Last run:** 7 chapters, 0 pages, 0 skipped
```

### Rules

- LIBRARY.md is the mandatory agent entry point — agents never glob through `library/`
- The Librarian updates LIBRARY.md at the end of every generate or update run
- Updates are always additive — existing book entries are preserved when a new book is added
- `**Books:** N` count and `**Last updated:**` date are updated on every Librarian run
- Foundational books appear in LIBRARY.md but are never generated or modified by the Librarian

## Naming Conventions

| Artifact | Convention | Examples |
|----------|-----------|---------|
| Book directories | kebab-case | `design-system/`, `playground-conventions/`, `component-library/` |
| Schema files | UPPER_CASE with .md extension | `BOOK.md`, `CHAPTER.md`, `PAGE.md`, `LIBRARY.md` |
| Chapter directories | kebab-case | `global-theme/`, `components/`, `typography/` |
| Page directories | kebab-case | `button/`, `input-field/`, `navigation-bar/` |
| Chapter/page IDs in frontmatter | kebab-case | `global-theme`, `button`, `input-field` |
