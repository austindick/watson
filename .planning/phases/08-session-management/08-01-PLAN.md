   - Ask prototype name as plain text question (NOT AskUserQuestion) — name only
   - Derive slug: kebab-case (`toLowerCase().replace(/\s+/g, '-').replace(/[^a-z0-9-]/g, '')`)

   - Gather branch list: `git branch --list 'watson/*'` for local branches

     - "Reset inactivity timer": update STATUS.md `last_activity` field via Edit tool, return to branch list
     - "Continue working on it": proceed with switch
   - Batch cleanup: if 2+ inactive branches, show "Clean up all inactive" option in the branch list
   - On branch selection: check for uncommitted changes (`git status --porcelain`), auto-commit if needed (`watson: checkpoint`), then `git checkout watson/{slug}`
   - Health check: verify blueprint/ and STATUS.md exist on the branch; if missing, inform user and offer to re-scaffold via watson-init
   - Update /tmp/watson-active.json: add `"branch": "watson/{slug}"` via Edit tool
   - Load STATUS.md frontmatter, display context summary: "[name] — [N] section(s) built. [M] pending amendment(s). Last session: [date] by [user] — [summary]."
   - If `drafts:` non-empty, include pending amendments note (preserving Phase 7 behavior)

**Section A: "Branch Operations" (new utility section, after the Verification section)**

This section is referenced by SKILL.md for both new-prototype and continue-existing paths. It contains the detailed git mechanics that SKILL.md's fork section references but doesn't inline.

Contents:
1. **Auto-commit guard:** Check `git status --porcelain`. Only commit if non-empty: `git add -A && git commit -m "watson: checkpoint before switching to {slug}"`. Never use `--allow-empty`.

2. **Branch creation (new prototype):**
   - Receive `prototype_name` and `slug` as parameters from SKILL.md
   - Auto-commit guard (step 1)
   - `git checkout main && git pull origin main`
   - Check if `watson/{slug}` already exists: `git branch --list "watson/{slug}"`
   - If exists: AskUserQuestion — header: "Branch exists", question: "watson/{slug} already exists.", options: ["Switch to existing branch", "Create watson/{slug}-2"]. If switching, read STATUS.md from that branch via `git show watson/{slug}:blueprint/STATUS.md` and display prototype name for confirmation.
   - If not exists: `git checkout -b watson/{slug}`
   - After branch creation, commit initial blueprint: `git add blueprint/ && git commit -m "watson: initialize {prototype_name} blueprint"`
   - Update STATUS.md `branch:` field via Edit tool: replace `branch: ""` with `branch: "watson/{slug}"`
   - Update `/tmp/watson-active.json` via Edit tool: add `"branch": "watson/{slug}"` and `"actions": []`

3. **Collaboration fork (branching from another user's prototype):**
   - This is the ONE exception to "always branch from main"
   - Receive source branch name (e.g., `watson/their-slug`) as parameter
   - User provides their own prototype name (plain text) for the fork
   - Derive new slug from their name
   - `git checkout -b watson/{new-slug}` (branches from the source branch, inheriting all blueprint files)
   - Create new STATUS.md with current user as owner (use Setup Flow data)
   - Add `forked_from: "watson/{their-slug}"` and `forked_from_owner: "{their-username}"` to STATUS.md frontmatter via Edit tool
   - Commit: `git add blueprint/ && git commit -m "watson: fork {prototype_name} from watson/{their-slug}"`
