**Role:** One sentence — "Extract component interaction states from the design system library and structure discuss-provided context into a per-section INTERACTION.md."

**Critical Constraints:** (before execution steps, matching builder/reviewer pattern)
1. Output must match INTERACTION-EXAMPLE.md section headers exactly
2. Line budget: less than 50 lines — omit empty Tier 2 and Tier 3 tables (do not render them with "None" rows); User Flows and Responsive Behavior sections always present (with fallback text when empty)

4. No Figma calls — work entirely from LAYOUT.md, DESIGN.md, and library PAGE.md files

**Inputs:** (update from stub to include new params)
- nodeId — section nodeId (accepted per contract; not used by interaction agent in Phase 9 — reserved for Phase 10 parallel dispatch)
- sectionName — used to construct output path
- interactionContext — optional structured JSON object with keys: customStates[], flows[], transitions[], responsiveBehavior[]. null/absent signals library-defaults-only mode.
- crossSectionFlows — optional array of cross-section flow objects. Written to output only when present and non-empty. (Not used in per-section INTERACTION.md — passed through for consolidator.)
- blueprintPath — absolute path to the prototype's blueprint/ directory
- libraryPaths — string array of pre-resolved chapter/page file paths
- watsonMode — boolean; suppress interactive prompts when true
- layoutPath — path to section's LAYOUT.md (staging: .watson/sections/{sectionName}/LAYOUT.md)
- designPath — path to section's DESIGN.md (staging: .watson/sections/{sectionName}/DESIGN.md)

**Outputs:**
- .watson/sections/{sectionName}/INTERACTION.md — interaction spec (less than 50 lines)

**Execution — 5 steps:**

**Step 1: Read spec files and extract component names**
Read layoutPath (LAYOUT.md) and designPath (DESIGN.md) for this section.
- From LAYOUT.md Component Tree block: extract component names — text before the first `(` on each indented line. Example: `└── AddToCartButton (Button, primary, full-width)` extracts "Button".
- From DESIGN.md Component Mapping table: extract Component column values, strip variant annotations in parentheses. Example: `Button (primary)` extracts "Button".
- Union both lists and deduplicate.
- Normalize each name to kebab-case for CHAPTER.md lookup: insert hyphen before each uppercase letter, lowercase everything, strip leading hyphen. Handle special cases: "Input/Select" normalizes to "select", "IconButton" normalizes to "icon-button", "TextInput" normalizes to "text-input".

**Step 2: Resolve library PAGE.md paths and read states**
Read the components CHAPTER.md from libraryPaths (the path ending in `components/CHAPTER.md`). Parse the `pages:` YAML frontmatter block to get id-to-path mappings.
For each normalized component name from Step 1:
- Find matching id in CHAPTER.md pages
- If found: read the corresponding PAGE.md, extract the `## States` section bullet list. Store as component states.
- If NOT found: mark as "net-new" for Tier 3

**Step 3: Build Tier 1 table**
For each matched component: create one row with:
- Component display name (original PascalCase with variant annotation from LAYOUT/DESIGN, e.g., "Button (primary)")
- States: comma-separated list from PAGE.md ## States bullets
- Notes: a brief contextual annotation (one clause, not a sentence) explaining how the component is used in this section — infer from the component's position in LAYOUT.md component tree (e.g., "Add-to-cart action", "Sort dropdown", "Status indicator — static")

**Step 4: Handle interactionContext**
Check: `if (!interactionContext)` — treat null AND absent identically.

If interactionContext IS provided (non-null):
- Map customStates[] entries: check each element name against the DS components found in Step 2. If the element matches a DS component, place in Tier 2. If the element does NOT match any DS component, place in Tier 3 as a net-new interaction.
- Map transitions[] → Transitions table rows (columns: Trigger, From, To, Animation, Duration)
- Map flows[] → User Flows subsections. Each flow gets a ### heading with the flow name, followed by a code block with indented step sequence (compact — no blank lines between steps)
- Map responsiveBehavior[] → Responsive Behavior table rows (columns: Breakpoint, Behavior)

If interactionContext is null/absent (fallback mode — INTR-04):
- Tier 1 table populated from Step 3 (library defaults)
- Add blockquote header note at top: "> No custom interaction context provided — library component defaults only"
- Tier 2: write "None — no custom interaction context provided." (do NOT render a table)
- Tier 3: write "None — no custom interaction context provided." (do NOT render a table). Exception: if Step 2 found components not in the DS, list them in Tier 3 as net-new even in fallback mode.
- Transitions: "None specified."
- User Flows: "No custom user flows specified — builder should implement standard component interactions only."
- Responsive Behavior: "No responsive behavior specified — builder should follow standard DS responsive patterns."

**Step 5: Write INTERACTION.md**
Write to `.watson/sections/{sectionName}/INTERACTION.md` using the Write tool.
Structure (section headers must match INTERACTION-EXAMPLE.md exactly):
```
# INTERACTION: {sectionName}

[optional blockquote header note for fallback mode]

## States

### Tier 1: Design System States

[description line]

| Component | States | Notes |
|-----------|--------|-------|
[rows from Step 3]

### Tier 2: Custom States
[Tier 2 content or "None" text]

### Tier 3: Net-New Interactions
[Tier 3 content or "None" text]

---

## Transitions
[table or "None specified."]

---

## User Flows
[flow subsections or fallback text]

---

## Responsive Behavior
[table or fallback text]
```

Enforce less than 50 line budget: Tier 1 Notes column must be concise (one clause). Flows use compact indented format. If still over: omit Transitions table when all transitions are standard DS hover/focus.

After writing, verify the file exists and is under 50 lines. If over, trim Notes column entries and re-write.

Add explanatory text below the schema:
- `interactionContext` is populated when the conversation covered interaction specifics for that section. null/absent when interactions were not discussed for that section — signals library-defaults-only to the interaction agent.

- Discuss pre-categorizes interaction context into the four keys (customStates, flows, transitions, responsiveBehavior) during the conversation. The interaction agent maps these directly to INTERACTION.md sections without re-categorization.
